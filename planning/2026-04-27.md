# Planning Session — 2026-04-27

## Context

Two things happened this session, in roughly this order:

1. A long conversation that **reframed Poly's purpose**. Started from "context engineering should be co-equal with state management," moved through several intermediate framings (agent-as-primary-user, Prolog-flavored knowledge base, React-like reactive runtime over a typed schema), and landed on:
   > Poly = a typed live knowledge base for agentic systems. Polynomial-functor schema declares the *space of allowable systems and evolutions*; agents operate freely within that space; the runtime mechanically enforces the boundary. The schema, the live configuration, and the trajectory are three layers of the same artifact.

   Project mode is now explicitly **exploratory research, not product development**. This dropped a lot of self-imposed shipping pressure and let the design breathe.

2. **A v0 engine landed.** First real working code in this iteration of the project: a parser for an implemented subset of Poly, a typed model, two of three planned queries, `Display` impls that mirror source syntax, and example files validating both shapes (interfaces with and without state-machine sugar).

## Decisions Made

### 1. Active surface is `src/engine/`

Replaces the older `src/diagram/` line as the path for new work. The `diagram*` modules contain useful algebraic primitives (terms, transforms, paths, BFS query) but a different concrete representation than the engine needs (named interfaces and lenses by name). Kept around for reference.

### 2. State-machine sugar uses the universal state machine, not `p ∘ p`

Earlier (2026-04-03) note suggested desugaring `interface Foo` with transitions into a `Foo → Foo ∘ Foo` defer (cofree-comonad-flavored). Replaced with a simpler and arguably more useful desugaring:

For `interface Foo` over states `S = {A, B, ...}` with declared transitions `a -> dest`, produce three declarations:

- **`Foo`** — external interface. Positions = states; directions = declared actions (transitions stripped).
- **`Foo.internal`** — the universal state machine on `S`. Same positions; at each state `s`, directions are `{s=>t : t ∈ S}`. Polynomially this is `S · y^S`.
- **`Foo.run : Foo.internal -> Foo`** — the realization defer. Identity on positions. At each `s`, each declared action `a -> dest` is realized as the abstract transition `s=>dest`.

The space of all defers `Foo.internal → Foo` (with `Foo.internal` fixed) is the space of state-machine structures on `S`. The user's interface declaration picks out one. This is an explicit categorical interpretation of "structural possibility space + concrete realization."

A direct consequence worth surfacing later: directions of `Foo.internal[s]` that have *no* preimage under `Foo.run`'s `dir_map[s]` are *unrealized transitions* — structurally possible but unbound by any declared action. A natural future query.

### 3. Naming convention for sugar-derived declarations

- External: `Foo` (user's name, unchanged).
- Internal universal state machine: `Foo.internal`.
- Realization defer: `Foo.run`.

Considered and rejected: `Foo_step`, `Foo.step` (unclear what "step" meant), `Foo.observe`, `Foo.dynamics`, `Foo.project`. `Foo.run` won on aesthetic grounds.

Dotted names are stored as plain strings. The current `text::ident()` parser doesn't accept dots, so user-written defer references can't yet point at `Foo.internal`. Not blocking — autoderived names aren't user-written today.

### 4. Implemented language subset is now meaningful

The parser handles: line comments (`# ...`), `interface` (with or without transitions), `defer`. Sufficient to fully parse `test2.poly` and `graph.poly`, both of which now live in `examples/implemented/`. Other example files remain in `examples/` as design sketches — they use parameters, guards, schemas, agents, Kleene, etc., which are not implemented.

### 5. Two queries implemented; third deferred

- **Q1 (`explain_position`)**: given `(interface, position)`, project across every defer touching the interface — forward (image position) and backward (preimage set), with direction correspondences.
- **Q3 (`locate_action`)**: every `(interface, position)` where a given action name is a direction.
- **Q2** ("given position and action, what's next?"): not implemented as a separate method. With state-machine sugar in place, the answer is recoverable from `Foo.run`'s `dir_map` plus `pos_map`, but a dedicated query would be friendlier.
- A unified pattern-matching / unification-style query was discussed (Datalog-flavored). Deferred until the explicit queries make the needs concrete.

### 6. `Display` impls mirror source syntax

`Interface` and `Defer` print in the same shape they'd be written by hand. Test2.poly's printout is byte-identical to the source. Graph.poly's desugared output (external + internal + run) reads cleanly and shows the realization map directly.

This matters because: if a Poly artifact is going to serve as both the human-readable view of a live system and a parseable schema, round-trippability between text and runtime is foundational. Display being source-mirroring is the first piece of that.

### 7. Project mode is exploratory research

Saved as a memory (`feedback_research_mode.md`). Implications:

- Restarts and rethinks are normal, not failures.
- Agent practice projects are a *source of intuition*, not a curriculum to complete.
- High-value artifact is often *prose-with-math-sketches in `planning/`*, not code.
- Don't optimize for human writability; agents are a co-equal user.

## Action Items

### Near-term (next session or two)

- [ ] **Implement Q2** (`next_position(interface, position, action)`) directly. With sugar in place this is a small wrapper: forward through `Foo.run` to find the realized transition, project to `dest`. Useful even if the unification query eventually subsumes it.
- [ ] **Add the "unrealized transitions" query.** For an interface `Foo`, list every direction of `Foo.internal[s]` that has no preimage in `Foo.run`'s `dir_map[s]`. This makes the "structural possibility minus actualization" gap visible at the API.
- [ ] **Pretty-print path separator**: `Graph.internal.A=>A` has dots playing two roles (interface separator + ident-with-dots). Pick a different display separator for `interface.position` paths — maybe `Graph.internal:A=>A` or `Graph.internal/A=>A`. Internal storage stays a tuple; just the formatter changes.

### Medium-term

- [ ] **Parameterized positions** (`Count[n: Int]`, `Cell[i: Int, j: Int]`). Touches parser, the typed model, and queries. Decide whether parameters live as opaque payloads or whether the engine reasons about them (probably opaque to start).
- [ ] **Guards** (`if (n > 0)`, `(n > 0)` prefix). Probably opaque predicates initially — store them as expression strings, evaluate later when the engine has a runtime.
- [ ] **Allow dotted identifiers in defer references**, so user code can mention sugar-derived names like `Foo.internal`. Small parser change.
- [ ] **`schema` declarations** for inert algebraic data. Parsing only at first; not used in queries until parameters exist.

### Longer-term / open

- [ ] **Unified query language.** Once Q1/Q2/Q3 plus the unrealized-transitions query are in use, the shapes of "useful queries" should be clearer. Then design a single pattern/unification query that subsumes them. Datalog over polynomial structure is the leading framing.
- [ ] **The "configuration" layer.** Right now we have schema (the parsed `.poly`). The live runtime — agents currently in particular states, defer wiring possibly evolving — doesn't exist yet. This is where the React-reconciler analogy starts to bite: a live store, a query surface, a renderer that produces a `.poly`-shaped human view.
- [ ] **Trajectory layer.** Append-only log of accepted state changes. Replayable. Enables historical queries.
- [ ] **Constraint enforcement.** When the live layer exists, the runtime should reject schema-violating proposed changes. This is where "bounded latitude" becomes operational.
- [ ] **Org-structure rendering** (the React-analogy direction). Not before the live layer exists, but worth holding as the eventual shape of the operator-facing surface.
- [ ] **Decide what to do with `examples/01_tool_use/`** (the exploratory `tool` / `agent` block sketches). Likely to evolve under the new framing — `tool` and `agent` may stay, but their semantics will be reread as "interfaces with particular runtime affordances," not new keywords.

### Not action items, but worth keeping visible

- The data-structures-as-interfaces vs. as-parameter-primitives question (open since 2026-04-03) is still open. Resolve via examples once parameters are implemented.
- `logru` may become load-bearing if Poly grows a Datalog query layer. Keep it in scope; don't churn the dependency.
- Parser currently stops at the dot in identifiers; the dotted names we generate are storage-only. Worth extending the identifier grammar before users start writing code that needs to reference sugar-derived names.

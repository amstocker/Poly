# Planning Session — 2026-04-03

## Context

Poly is a language/runtime for managing the composition of non-deterministic agent choices with deterministic user flows, grounded in the mathematics of polynomial functors (https://arxiv.org/pdf/2312.00990).

This session focused on solidifying the core language design before implementation.

---

## Decisions Made

### 1. Poly will be a custom language, not a configuration format

Considered using an existing config language (TOML, YAML, etc.) for better LLM compatibility, but decided against it. The polynomial functor concepts (interfaces, defers, guards, parameterized states) don't map naturally onto config languages and would be verbose and awkward. Modern LLMs handle DSLs well given a few examples, and the existing `.poly` syntax is already intuitive enough to be learnable from a short system prompt.

### 2. Two core constructs: `interface` and `defer`

- **`interface`** declares a polynomial functor: a set of states, each with a set of actions.
- **`defer`** declares a lens (natural transformation) between two interfaces.

`view` is deprecated — it was an earlier keyword for the same concept. `defer` is preferred because it captures the runtime semantics: "this interface defers to that interface for decisions."

### 3. Defer semantics (the lens structure)

A `defer A -> B` consists of:
- **Forward map**: a total function from states of A to states of B.
- **Backward map** (dependent on forward): for each state mapping `a -> b`, a function from B's actions at `b` back to A's actions at `a`.

Concrete example from `test2.poly`:
```
interface Test:  A { B, C }, E { F }
interface Test2: W { X, Y, Z }

defer TestDefer : Test -> Test2
    A -> W { X | Y -> B, Z -> C },
    E -> W { X | Y | Z -> F }
```
Forward: `A -> W`, `E -> W`. Backward at `A -> W`: `{X,Y,Z} -> {B,C}`.

### 4. Partial defers (from a sub-interface)

A defer does not need to cover all states of the source interface. Syntax like `Board.Idle -> Prioritizer` defers only from the `Idle` summand of Board's polynomial. This avoids polluting target interfaces with irrelevant passthrough states. Important to account for in the engine design.

### 5. Dynamic interfaces (comonads / state machine sugar)

Inline transitions in an interface:
```
interface Counter
    Count[n: Int] {
        Increment -> Count[n + 1],
        (n > 0) Decrement -> Count[n - 1]
    }
```
...are syntactic sugar that actually defines three things:

1. **An anonymous bare interface** (the underlying polynomial, states + actions, no dynamics):
   ```
   interface _
       Count[n: Int] { Increment, (n > 0) Decrement }
   ```
2. **The comonad `Counter`** — the bare interface equipped with the comultiplication (i.e., the full state machine).
3. **The comultiplication defer `Counter.step`** — the lens from `Counter` to `Counter . Counter` that encodes the transition function.

The name `Counter` refers to the comonad (the whole package), not just the bare polynomial. The bare polynomial is an implementation detail, though it may be useful to reference explicitly in some cases (e.g., deferring to just the bare interface without dynamics).

### 6. Everything should be named

All defers (including sugar-derived ones) should have names for tracing and debugging purposes.

### 7. Agents are just interfaces

An agent (LLM, human, external process) is represented as an ordinary interface. The `defer` wires it up identically to any other interface. What makes it an "agent" is a deployment concern (who fills the interface at runtime), not a language concern.

### 8. Primary authoring story

A combination of human and LLM authoring. Humans write the initial structure; agents may author or modify parts. Future possibility of hierarchical mutability (only certain parts of a script are agent-mutable).

### 9. Schemas for structured data

`schema` is a third top-level construct (alongside `interface` and `defer`) for defining passive data types that parameterize states and actions.

```
schema TaskData
    title: String,
    description: String,
    priority: Priority

schema Priority
    High, Medium, Low
```

Key properties:
- **Product types** (named fields): `schema TaskData { title: String, ... }`
- **Sum types** (enums): `schema Priority { High, Medium, Low }`
- **Generics**: `schema Result[T] { Ok[value: T], Err[message: String] }`
- **No recursion** — recursive/navigable structures (trees, linked lists) belong as polynomial interfaces, not schemas
- **Built-in primitives**: `Int`, `Float`, `String`, `Bool`
- **Built-in collection types** (`List[T]`, `Map[K, V]`, etc.) are technically interfaces (cofree comonads) with convenient built-in syntax, not schemas

Schemas appear in three places:
1. **State parameters**: `Pending[request: RequestData]`
2. **Action parameters**: `Fetch[city: String, options: QueryOptions]`
3. **Guards**: `if (task.priority is High)`

The distinction from interfaces: schemas are *inert, non-recursive data* (no states, no actions, no dynamics). Interfaces are *interactive and potentially recursive*. Anything that needs to be navigated, iterated, or has inherent structure beyond flat products/sums belongs as an interface.

### 10. The Poly engine is reactive, not imperative

The engine works like a logic engine (similar to Prolog). A `.poly` script configures the engine so it can infer state transitions based on actions. The engine holds the current state of all interfaces, and when an action arrives (from an agent, human, or external system), it propagates the resulting transitions through the defer graph.

Async is not a language primitive — it's just states. An external LLM interface in `Generating` state is no different from any other state; the host knows it means "waiting for a response" and supplies the action when it arrives. The Poly script doesn't know or care about timing.

### 11. External interfaces

The `external` keyword marks interfaces whose state is owned by the host, not by Poly. The polynomial structure is identical — same states, actions, and defer semantics. The difference is purely at the runtime level: the host fulfills the interface by supplying actions when external events occur.

```
external interface LLM
    Ready { Prompt[messages: List[Message]] },
    Generating { Complete[response: String], Fail[error: String] }
```

This keeps the language pure and pushes all runtime/platform concerns to the host.

## Open Questions

### 1. Syntax for polynomial composition

The comonad comultiplication requires `p . p` (composition of a polynomial with itself). No syntax has been settled on. Dot notation (`Counter . Counter`) is one option (Haskell-like), but the interior of a defer block over a composed polynomial also needs syntax. May be too complex to expose directly — could remain hidden inside the state machine sugar.

### 2. Data structures: interfaces vs. built-in parameter primitives

Data structures (lists, maps, stacks, priority queues) are needed for real systems. Three approaches considered:

- **As parameters**: States are parameterized over data (`Board[tasks: Map[Id, Task]]`). Requires an embedded expression language for guards *and* for parameter transformations in transitions (`items.push(task)`, `items.pop`).
- **As interfaces**: Data structures are polynomial interfaces (e.g., `List` is the cofree comonad of `Ay + 1`). Mathematically uniform but verbose and conceptually heavy for users.
- **Middle ground**: Built-in primitives (`Stack[T]`, `List[T]`) that are sugar for underlying interface definitions. Users interact with them as primitives but they're interfaces under the hood.

The middle ground is promising but not yet validated with examples.

### 3. Parameter mutation

When a state transition changes parameterized data (e.g., pushing to a stack), how is that expressed? Options:
- Explicit transformation expressions in transitions: `Push[t] -> Idle[items.push(t)]`
- Through defers to the data structure's own interface
- Implicitly via comonad structure

This is closely tied to the data structure question above.

### 4. Built-in collection interfaces

`List[T]`, `Map[K, V]`, etc. are interfaces (cofree comonads), not schemas. The exact syntax and semantics for interacting with them (both in state parameters and through defers) hasn't been worked out yet. Need examples showing how agents navigate and mutate collections through the interface mechanism rather than through parameter expressions.

### 5. Multiple agents deferring to the same structure

The `WorkQueue` example showed two agents (planner pushing, worker popping) interacting with the same data. How are concurrent/conflicting interactions resolved? This is a runtime/engine question more than a language question, but the language may need to express ordering or priority.

## Examples Sketched

- **Task management system**: `Task` lifecycle interface, `Board` managing multiple tasks, `Prioritizer` and `Router` agents as interfaces, defers wiring them together.
- **External data source**: `external interface TaskDB` with query/results/focus states.
- **Shared work queue**: `WorkQueue[items: Stack[Task]]` with push/pop, deferred to by planner and worker agents.

- **External interfaces**: `external interface WeatherAPI` and `external interface LLM` showing how async external systems are just ordinary interfaces whose state is fulfilled by the host.
- **Schemas in context**: `TaskData`, `Priority`, `Assignment`, `Message` schemas parameterizing states and actions in the task management system.

These examples live in the conversation and should be formalized into `.poly` files as the design solidifies.
